律师的工作本质上是在**信息不对称**和**高风险**的环境中，通过**解决问题（Resolve Issues）**来达成**商业目的（Business Goal）**。

以下我将从律师的心智模型出发，深入拆解 Case 详情页的设计逻辑，并优化你的“Issue 驱动”工作流。

---

### 一、 律师的心智模型：他们在想什么？

在打开一个 Case 详情页的瞬间，律师的大脑中会并行处理三层信息：

1. **情境感知 (Context Awareness):** “这是哪个案子？截止日期是什么时候？我们在什么阶段？”（焦虑来源：Deadline）
2. **风险雷达 (Risk Radar):** “哪里出错了？有什么合规漏洞？文件质量有问题吗？”（职业本能：纠错）
3. **行动导向 (Action Oriented):** “我现在需要做什么？我需要给客户发什么邮件？怎么把这个案子推到下一步？”（价值产出：解决问题）

**你的假设验证：**
_“以 Issue 为线索，包含 QC 和 Compliance Check”_ 完美击中了第 2 点（风险雷达）和第 3 点（行动导向）。**律师不希望看到一个静态的“文件仓库”，他们希望看到一个动态的“待办/风险列表”。**

---

### 二、 Case 详情页的主体区域设计策略

#### 1. 核心布局建议 (The Layout)

我们将主体区域划分为三个逻辑区块，以支撑律师的认知流：

- **顶部锚点区 (The Anchor - Context):**
  保持不变

- **左侧/中央工作区 (The Evidence - The "What"):**
- **内容：** **Issue List (你的核心线索)**。这里不仅仅是列表，而是“风险控制塔”。
- **包含：** 自动化的 QC 结果、合规引擎跑出的 Red Flags、人工标注的风险点。

#### 2. “以 Issue 为线索”的深度串联逻辑

你提到的 _QC（质量检查）_ 和 _Compliance Check（合规检查）_ 是 Issue 的**来源（Source）**，而**引导客户沟通**是 Issue 的**出口（Resolution）**。

我们需要构建一个 **"Issue Lifecycle" (问题生命周期)** 视图：

**阶段 A: 发现与聚合 (Detection & Aggregation)**

- **不仅是报错：** 不要把 QC 和 Compliance 做成两个独立的 Tab。律师并不关心是“格式错误”还是“合规风险”，他们关心的是“阻碍结案的问题”。
- **设计建议：** 将所有机器发现的问题（如：错别字、条款缺失、高风险实体）和人工发现的问题，统一归集到一个 **"Issues"** 中。

**阶段 B: 律师研判 (Triage & Lawyer Review)**

- 这是体现律师专业价值的环节。机器发现的 issue 可能是误报，或者是商业上可接受的风险。
- **操作流：** 律师对每一条 Issue 进行操作 —— `Ignore` (忽略), `Fix Internally` (内部修正), 或 `Mark for Client` (需客户确认)。

**阶段 C: 转化为沟通 (Conversion to Communication)**

- **这是你想法中最精彩的部分。** 很多系统止步于发现问题，而你的目标是引导沟通。
- **设计建议：** 在 Issue 列表中，允许律师勾选多个 Issue，点击 **"Draft Client Note"**。
- **自动化魔法：** 系统自动生成一封邮件草稿或一份《法律意见书》附件，将勾选的技术性 Issue 转化为通俗的客户语言，这里可以使用 email 或者 whatsapp 转发形式

---

### 三、 优化后的工作流线索 (Workflow Thread)

我们可以把这个工作流定义为 **"Identify - Triage - Resolve" (I-T-R 模型)**。

1. **Input (输入):** 上传文档/案件材料。
2. **Auto-Scan (自动扫描 - 你的 QC & Compliance):**

- 系统后台运行，生成初始的 Issue List。
- _用户体验：_ 进度条完成后，Issue 数量徽标亮起（例如：🔴 3 个严重风险）。

3. **Review (律师审查 - 核心交互区):**

- 律师点击 Issue，主视图自动跳转到文档对应的条款/位置（Contextual Linking）。
- 律师判断：这是真问题吗？
- 每一个 issue 都需要有一个 reference tag 去索引到 verified documents，更进一步地，也可以看 original evidence，都是环环相扣的。

4. **Action (处置):**

- 如果是**文档质量问题 (QC)** -> 律师直接在 UI 修复
- 如果是**实质合规问题 (Compliance)** -> 律师标记为“需与客户讨论”。

1. **Output (输出 - 客户沟通):**

- 将所有标记为“需讨论”的 Issues，一键导出为 **"Issue List / Questionaire"** 发送给客户。

### 四、 专家视角的补充建议

为了让这个模型更丰满，建议考虑以下两点：

**1. Issue 的“语境化” (Contextualization)**
律师非常讨厌脱离文档看 Issue。

- **Bad Design:** 一个单独的列表页写着“第 5 页有错别字”。
- **Good Design:** 分屏模式。左边是文档，右边是 Issue 卡片。点击卡片，文档自动滚动并高亮对应段落。

**2. 区分“内部 Issue”与“外部 Issue”**

- **内部 Issue (QC):** 格式、引用错误、错别字。这些是律师的“体面”问题，通常在这个阶段就必须消化掉，**不要**展示给客户看，否则显得不专业。
- **外部 Issue (Compliance/Commercial):** 风险条款、缺失证照。这些才是引导客户沟通的筹码。
- **策略：** 在 UI 上明确区分这两个 View，或者提供过滤器。

### 总结

你的想法 **“以 Issue 为线索”** 是非常符合现代法律服务流程（ALSP / Legal Ops）的高级思路。

**Case 详情页的核心叙事应该是：**

> "我们在这个 Case 中发现了 **[X]** 个问题，其中 **[Y]** 个是低级错误（已自动修复/内部消化），剩余 **[Z]** 个是关键风险，需要你（律师）立刻决策并生成给客户的**沟通清单**。"

这种设计不仅仅是在做管理软件，而是在做**律师的智能副驾驶（Co-pilot）**。
